# /iteration-design — 反復設計(幹＋ベット)

要件が動く現場の定常運用。立ち上げ(`requirements-definition` / `basic-design`)は済み、幹(現行仕様)が
薄く立っている前提で回す。目的は一つ:

> **人間が意図したこと ≠ AI が実際に作ったもの、という「認識のねじれ」を、溜めずに連続的に閉じること。**

`spec-recovery` は、このねじれを溜めてから一括で返済する**移行儀式**(遅行検出)。毎フェーズそれをやると
ベットに全額の儀式を払う羽目になる(→ 基本設計を書いて → 賭けが変わって → 最終作戦に書き直す、の churn)。
この skill は、儀式を確度に比例させ、ねじれを**出荷のたびに**閉じる。

## 第一規則: 儀式は確度に比例。だが出荷 delta の追認は確度に無関係で必須

**2 つの軸を混同しない**(混同が本番事故を生む — 未確定の賭けのコードを追認せず本番に出す事故)。

- **軸A(前向きの設計):** 確度 🔴🟡🟢 に**比例**して儀式を変える。賭けは薄く、殺せるように書く
- **軸B(出荷したコード):** 賭けの確度がどれだけ低くても、**merge した瞬間に必ず**その delta を幹に追認する

> 設計は安く賭けてよい。だが**理解しないコードは出荷しない**。軸B がねじれ除去の本体。

## 確度と投資の対応

| 確度 | 前向きの投資(軸A) | 幹での権威 |
|---|---|---|
| 🔴 OPEN | 課題管理表に **1 行**。設計文書は書かない | なし(未決の可視化そのもの) |
| 🟡 暫定(賭け) | **1 ページのベット票**。捨てる前提・born-未追認・現行仕様と別 dir に隔離 | なし |
| 🟢 確定(出荷して真) | 現行仕様の**該当セクションだけ**を born-追認で増分改訂 | 唯一の正 |

> **🟡→🟢 を動かすのは merge ではなく実測。** merge で証明されるのは「AI が意図どおり作った」(内部妥当性)
> だけ。「その賭けが顧客の問題を解いた」(外部妥当性)は、本番で実測が棄却条件に照らされて初めて決まる(④b)。
> 幹は merge 時に「システムが**何をするか**(事実)」を記録するが、賭けの確度が 🟢 に上がるのは**実測の後**。
> 出荷済み・幹に載っている・でも賭けは未証明(🟡)、という状態はふつうにあり得る。

🟡 を 1 ページに抑えるのは倹約だけが理由ではない。30 ファイルの設計を書くと賭けを殺すのが高くつく
(サンクコスト)。**薄い賭けは殺せる。**スタートアップの速度は「良い賭けを速く書く」ではなく
「悪い賭けを速く殺す」能力で決まる。

## ループ(定常・連続)

### ① ベット(🟡 ベット票 — **AI と壁打ちで起票**)
リードが 🔴 を一つ選ぶ。ベット票は**壁打ちで作る** — 課題が渡されていれば AI が課題＋事実(実測)から
**6 欄の草案を埋めて提示**する(空欄で丸投げしない)。リードは壁打ちで詰め(枠が違えば差し替え・棄却条件を鋭く・
測定の場所を具体に)、**最後にリードが明示的に確定(born-追認)する**。ねじれ防止の砦は「人間が孤独に全文を書くこと」
ではなく、**AI 草案のまま無追認で流さないこと** — 壁打ちはサイレントの逆(明示的な提案と決定)なので、
看板どおり「AI との壁打ち」で回してよい。中身は 6 欄:

1. **解こうとしている問題**(なぜ今これか)
2. **今回の賭け**(採る方針。1 つ)
3. **棄却条件**(何が観測されたら、この賭けは間違いか — 先に書く)
4. **測定の場所**(棄却条件を*いつ・どの実測で*測るか。どの本番指標・ログ・利用データ・実挙動を、
   出荷後どのタイミングで見るか。**ここを書かないとベットの証明が永久に来ない** — 賭けが 🟡 のまま漂う)
5. **appetite**(これに払う時間の上限。見積りではなく「いくらの価値があるか」)
6. **実装者が決めてはいけない 1 点**(= 幹に触れる不可逆。ここは②で自分が決める)

置き場所は現行仕様と**別ディレクトリ**(例 `docs/ベット/` )。前向きの未追認設計が幹に混ざると、
メンバーの AI が古い賭けから再導出を始める。隔離は語彙単位で機械化してよい(bootstrap 圏外の
repo-local テスト = docs-coherence 方式。覆った語彙が幹に残ると落とす)。

### ② 発注(作業依頼書 — 既知の空白に名前を付ける)
賭けを 1〜N 個の作業依頼書に割る(様式は `templates/docs/基本設計/作業依頼書.md`)。
**「あなたが決めること」欄が、この skill のねじれ対策の中核**: AI(や人間メンバー)に委譲する空白を
*明示的に列挙*することで、silent な空白を **known** に変える(= 不変条件「未決は観測される」の実装)。

- **前提**欄 = 幹の**確定セクションへの参照**で埋める(転記でなく参照。幹が動けば発注も追従)
- メンバー(とそのAI)が受け取るのは「幹の該当セクション + この作業依頼書 1 枚」。**賭けの束は渡さない**

ここまでが upstream の領域。**③ 以降の実装に、この skill は関与しない**(発注文を書き終えた時点で手を離す)。

### ③ 実装(この skill は触らない)
実装のフローは現場のもの。**bootstrap を採用していれば** TDD 先行・arch 依存方向・lint・sprint 並列レーン・
verification が hook で強制される。**採用していなくてもこのループは回る** — この skill は bootstrap に依存しない
(下記「非依存」)。渡す境界は作業依頼書まで。

### ④a 継ぎ目で追認 — ねじれ除去(merge 時・事実の記録)
merge / 完了の瞬間、**AI が実際に作った delta を人間が読み**、現行仕様の該当セクションに写す。
ここで閉じるのは**内部妥当性**(AI が作ったもの = 私が意図したもの)**だけ** — 賭けが正しかったかはまだ未証明:

- delta を読み、**生成の妥当性でなく意図との一致**を判定する(spec-recovery の仕分けと同じ手)
- 現行仕様の該当セクションを born-追認で増分改訂する。署名は**見出しに 1 個**
  (`## 見出し 【確定 YYYY-MM-DD 本人追認】`)。文単位の署名増殖はしない(spec-recovery の規則を継承)
- ここで幹に載るのは「システムが**何をするか**」= 事実。*良し悪し*は載せない。この時点で賭けは **🟡 のまま**
- **幹が育つのは、ここで追認された delta によってのみ。**一括再導出は二度としない

### ④b 棄却条件の評価 — ベットの証明(本番・実測・appetite 期限)
出荷後、①④の**測定の場所**で実測(利用・データ・挙動)が集まったら、**先に書いた棄却条件に照らす**。
これが 🟡 を動かす唯一の点 — **merge ではなく実測が確度を上げる**:

- **棄却条件がヒット** → 反証。⑤ 回路遮断へ(賭けを殺す)
- **満たさず、正のシグナルが出た** → **🟢 に昇格**(= ベットが証明された。幹の該当セクションの確度印を上げる)
- **appetite を超えてもどちらも出ない** → 宙吊りを許さず**再判断を強制**(Shape Up の circuit breaker)
- 実測を*集める*計装は bootstrap の verification(HUMAN 行「これは顧客の欲しかった物か」)と噛む。
  だが**実測を棄却条件に照らす判断は upstream の領域**(妥当性の判定は人間)

### ⑤ 回路遮断(賭けを殺す)
④b で棄却条件がヒット(または appetite 超過で正のシグナルも無い)なら、**賭けを殺す**。🟡 ベット票は破棄する
(`archive/` に退避しない — **アーカイブは git 履歴そのもの**)。幹には**何も畳まない**。
出荷しなかった賭けはコードにならなかったので、そこに理解負債(ねじれ)は存在しない
(作らなかったシステムの理解は誰も負わない)。**殺すのは失敗ではなく設計どおりの動作。**

## ベット並行数 = 追認キャパシティ(Kanban WIP)

同時に開いてよい 🟡 の数を絞る。律速は**リードが④a の継ぎ目で読み切れる量**(オラクルが 1 人なら
それが制約)。スプリントの代わりにこれで流量を制御する。

> **注意(bootstrap との相性):** これは `.bootstrap/wip`(実装レーンの並列数)とは**別物**。
> 1 つの賭け → 複数の作業依頼書 → 複数のレーン、になりうる。ベット並行(この skill・製品スコープ)と
> レーン並行(bootstrap・実装並列)を同じ数で縛らない。

## 形にしか出ない空白(delta 追認が取りこぼす種類)

理由なき NULL 許容・誰も選んでいない enum の枝・黙って方針になった既定値 — これらは delta に現れても
目立たず、④で skim され取りこぼされる。これを回収するのが `spec-recovery` の本来の役割:
**再構築ではなく、幹に対するスコープ付き drift 監査**。

- 発火信号(現行仕様が最後に追認されてから幹に触れる migration が N 件、等)は、**手動でも repo-local な
  仕組みでも起こせる**。bootstrap を使う現場なら doctor が surface して**くれてもよい**(任意の上乗せで
  あって前提ではない)。いずれにせよ**「これは欠陥か仕様か」の判断は upstream の領域** — 信号で `spec-recovery`
  を有界監査としてロードし、コード＋**実測**と幹を突合し、**食い違いだけ**を課題管理表に起票する(全 wipe ではない)
- bootstrap がある現場では、実装の継ぎ目での backstop(データ書込み時の「修復か仕様か」可視化 = ADR 0013)も
  上乗せされる。だがそれは保険であって、ねじれ除去の本体はこの監査(=判断)にある

## 課題管理表(🔴)と ADR(🟢)のライフサイクル

決定の家は 2 つだが競合しない。**同じ決定の別段階**:

- **未決(🔴)** → 課題管理表の 1 行(bootstrap 採用 repo でも `docs/decisions/` とは別に持つ)
- **確定した不可逆(🟢)** → born-追認の ADR(`docs/decisions/`)。**merge は署名だが理解ではない** —
  読まずに merge された AI 起草 ADR は【doc主張】のまま権威を持たない(born-追認 が bootstrap の ADR も統べる)

## bootstrap との分界(要約)

| | upstream(この skill) | bootstrap |
|---|---|---|
| 領域 | 設計・計画・チーム運用・**認識のねじれの除去** | 実装・統合の**強制**と実装時の可視化 |
| 手段 | 手順+成果物(判断を支える) | hook(前提を fail-closed で課す・判断は surface) |
| 渡す境界 | 作業依頼書を書くまで | 作業依頼書を受けてから |
| ねじれへの関与 | **本体**(①意図の明示・②既知空白の命名・④充填の追認・drift 監査の判断) | backstop(実装の継ぎ目での可視化・merge gate) |

> **非依存(重要):** この skill は bootstrap に**依存しない**。使わない現場でも単体で回る(①②④は手順と成果物だけで
> 成立する)。bootstrap があれば③の強制と信号 surface が**上乗せ**されるだけ。逆に bootstrap も upstream を要求
> しない。2 プラグインは片方だけでも成立する疎結合 — 依存関係(片方が無いと動かない)は作らない。

## 完了判定(1 ループ)

- [ ] 🟡 ベット票に 6 欄(特に**棄却条件**・**測定の場所**・**実装者が決めてはいけない 1 点**)が揃っているか
- [ ] 作業依頼書の「あなたが決めること」で、委譲した空白に名前が付いているか
- [ ] 出荷した delta が幹の該当セクションに born-追認で写ったか(④a・軸B。確度に関わらず必須)
- [ ] 🟡→🟢 の昇格が **merge でなく実測(④b の棄却条件評価)** を通ったか(merge だけで 🟢 を名乗っていないか)
- [ ] 殺した賭けの 🟡 票を破棄したか(幹に畳んでいないか)
- [ ] 幹に**再構築**でなく**増分**だけが入ったか(全部入りの再生成を毎フェーズやっていないか)
