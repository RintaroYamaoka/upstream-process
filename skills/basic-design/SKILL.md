# /basic-design — 基本設計(外部設計)= 幹の初版を立てる

外から見える振る舞い — データ・画面・外部接続 — を設計し、**他人(人間メンバーとその AI)に渡せる状態**まで
持っていく。出力は**幹(現行仕様)の初版** = `docs/current-spec/` の文書一式(born-追認済みのものだけが幹)と、
委託単位ごとの作業依頼書(ベットフォルダ内)。

成果物の定義・完了基準は `references/artifact-catalog.md`、様式は `templates/docs/current-spec/` を参照する。
課題管理表(`docs/issue-log.md`)は要件定義から持ち越して使い続ける。

## 立ち上げの規律: 不可逆だけ固める(この工程は一度しか回さない)

- 固めるのは**不可逆の少数**: データ所有・スキーマの形・外部契約・中心仮説(先行試作)。
  **可逆の多数**(画面の詳細・機能の網羅・フロー)は 🔴 のまま課題管理表に残す — 揮発性を幹の外へ押し出す
- **フェーズごとにこの工程を回さない**(それが churn の元)。要件が動き始めたら定常の主役は
  `iteration-design`(幹＋ベット)で、以後の幹は出荷 delta の④a追認によってのみ育つ
- 出力先が独立の「基本設計」フォルダでなく `docs/current-spec/` なのは意図的: **幹の家は最初から一つ**。
  前向き設計と現行仕様の二本立ては drift の温床(AI が古い方を典拠に再導出する)。この工程で書く文書は
  「実装されればこうなる」の宣言であり、出荷のたびに④aの追認で現実へ追従していく

## born-追認(幹に載る条件)

文書は AI 起草でよいが、**セクション見出しに署名(`## 見出し 【確定 YYYY-MM-DD 本人追認】`)が入るまで草案**
であり、幹の権威を持たない。署名の等級(本人訂正 / 本人追認 / 本人指示(AI筆記))は spec-recovery の定義を継承する。

## 進め方(順序に意味がある)

### 1. システム構成図+データ所有者表
誰がどのデータを持ち、どう取得し、どれくらい新しいか。
**完了条件は「図が描けた」ではなく「全行に実物確認の日付が入った」。**
外部 API は 30 分叩く。実データは 1 件目で見る。確認できない行は課題管理表の「観測待ち」へ。
ここを図面だけで通過すると、実装後に本番データで初めて設計誤りが判明する(最も高い授業料)。

### 2. テーブル定義書(ER図)+ 用語集の昇格
用語集からエンティティを起こす。このとき用語集を `docs/requirements/glossary.md` から
`docs/current-spec/glossary.md` へ**昇格**(`git mv`)する — 実装時の型の枝は全部ここから生えるので、
以後は幹の一部として④aで改訂される(二本の正を残さない)。各カラムに 3 注記:
**保存か導出か**(導出できる値を保存するなら理由必須 — 保存した値は嘘をつける)/
**NULL の業務上の意味**(「無い」はいつ起こるか)/ **0 と「未観測」の区別**。
AI 起草の締めの申告(自分で決めた箇所の列挙)を必ず取る。テーブル設計は補完が最も混入しやすい。

### 3. 先行試作(プロトタイプ)= 最初のベット
**画面設計より先に**、一番細いデータ経路を 1 本通す(実データ → 中心の計算 → CLI か HTML 1 枚)。
UI・認証・管理画面は含まない。中心仮説はまさに賭けなので、**ベットとして起票する**:
`docs/bets/BET-01-<slug>/` にベット票(様式 = `templates/docs/bets/bet.md`。棄却条件 = 中心仮説が覆る観測、
appetite = 数日)。実装そのものは bootstrap の管轄で行う(この skill は発注文を書くまで)。
発注者・利用者に見せる。ここで中心仮説が覆った場合の被害はほぼゼロ。画面の後で覆ると全損。

### 4. 画面一覧・画面遷移図(先行試作が生き残ってから・立ち上げでは薄く)
各画面の目的を「誰が・何を・30秒以内にできる」で書く。全画面が機能一覧に紐づくことを確認。
**見た目・並び・余白は描き込まない** — それは実装担当者の決定領域。ここで描き込むと相手の決定権を先取りする。
画面は**可逆**なので立ち上げで全画面を網羅しない — 未設計の画面は 🔴 のまま課題管理表へ。

### 5. CRUD表
機能×テーブルの交差表。AI に埋めさせ、人間は交差チェックだけ読む:
どの機能からも触られないテーブル・どのテーブルも触らない機能が残っていたら、どちらかが発明か漏れ。

### 6. 外部インタフェース一覧
契約は相手の文書でなく**相手の実挙動**で確認する(実レスポンス・レート制限・認証の実確認日付)。

### 7. 責任分担表(RACI)
判断領域ごとの決定者。作業の分担ではなく**判断の分担**。
「その判断を間違えたとき、最初に気づけるのは誰か」が決定者。
安定した割当だけをここに書き、案件ごとの割当は作業依頼書の「あなたが決めること」欄が主。

### 8. 作業依頼書(委託単位ごと・ベットフォルダ内)
様式 `templates/docs/bets/work-order.md`。**配置はベットフォルダ**(`docs/bets/<BET-slug>/WO-XX-<slug>.md`)—
`docs/current-spec/` やフェーズフォルダに散らさない(iteration-design ②と同じ規則)。中核 4 節:
**前提(確定済み・変えない)・実現してほしい振る舞い(観測可能な文)・あなたが決めること・依存する未決課題(課題番号)**。

- **前提欄は幹の確定セクションへの参照で埋める(転記でなく参照)** — 幹が動けば発注も追従する。
  メンバー(とその AI)に渡すのは「幹の該当セクション + この依頼書 1 枚」。賭けの束は渡さない
- 「あなたが決めること」の空欄は「全部こちらが決めた」の意味になり、相手を AI と同じ位置(空白を無音で埋めるしかない位置)に置く
- 受領側の締め: 完了時に「依頼に書かれていなくて自分で決めたこと」を申告してもらい、申告欄に転記

## 変更管理(前提が覆るときの通知規律)

**前提を覆すとき**(用語の再定義・テーブル変更・機能の廃止):
1. その前提を引用している**進行中の作業依頼書**を洗う(前提節の参照で grep できる)
2. 依頼先に通知する
3. それから幹の該当セクションを born-追認で改訂する

承認済みの依頼の下から前提を黙って抜くと、相手の成果物が**相手の落ち度の顔をして**壊れる。
定常に入ってからの変更の**本経路**は iteration-design のループ(①〜⑤)であり、この通知規律は
そのループの②〜④aの中でも効き続ける。

## 工程完了判定

- [ ] データ所有者表・外部IF一覧の全行に実物確認の日付があるか
- [ ] テーブル定義の全カラムに 3 注記(保存か導出か / NULL の意味 / 0 の区別)があるか
- [ ] 用語集が `docs/current-spec/` に昇格済みか(`docs/requirements/` に残っていないか)
- [ ] 先行試作のベットは決着したか。中心仮説は生きているか
- [ ] CRUD 表の交差チェックは通ったか
- [ ] 幹に置いた全セクションに born-追認の署名が入ったか(署名の無い文書は草案 = 幹の権威を持たない)
- [ ] 全委託単位に作業依頼書(ベットフォルダ内)があり、「あなたが決めること」が埋まっているか
- [ ] 未決課題に依存する依頼は、課題番号が依頼書に明記されているか

ここから先(詳細設計・実装・テスト・統合)は bootstrap の管轄。要件が動き始めたら
`/upstream-process:iteration-design` へ — この skill を二度目に回すのではなく。
