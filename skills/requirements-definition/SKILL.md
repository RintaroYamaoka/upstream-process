# /requirements-definition — 要件定義

この skill は、実装に入る前に決めるべきことを古典の様式で決め切る(または未決として台帳に載せる)ためのもの。
コードは 1 行も書かない。この工程の出力は `docs/requirements/` の文書一式と課題管理表(`docs/issue-log.md`)である。

成果物の定義・完了基準は `references/artifact-catalog.md`、様式は `templates/docs/requirements/` を参照する。

## 立ち上げの規律: 薄く(全部入りを作り込まない)

要件が動く現場では、この工程で全成果物を完成させようとしない。目的は**不可逆の少数**
(スタック・コアドメイン・スキーマの形・外部契約)を見極める材料を揃えることで、薄いデフォルトは
**企画書・用語集の種・課題管理表**の 3 つ。残り(業務フローの詳細・機能の網羅・非機能の全項目)は
🔴 のまま課題管理表に載せる。**作らなかった成果物は「未作成」を課題管理表に 1 行**(欠落を無音にしない)。
古典どおり全部を作り込む価値があるのは、前提が安定した(要件がもう動かない)現場だけ。
確度の語彙は iteration-design と共通: 未決 = 🔴。

## 前提(skill 実行前に人間がやること)

**一次情報(ヒアリング記録・既存資料・実データ)は人間が全文読む。** AI の要約から入らない。
要約は語彙を平坦化し、業務の例外ケース・原文の言い回しが最初に消える。
(実例: 一次情報に「メール」と書いてあったのに、要約経由で作られた型からメールが欠落した。)
AI の役割は要約器ではなく質問生成器。

## 進め方

### 0. 材料集め
一次情報を `docs/requirements/materials/` に集める(ヒアリング記録・既存資料へのリンク・実データのサンプル)。
無いなら「一次情報が無い」こと自体を課題管理表の 1 行目に起票する(推測で先に進んだ記録が残る)。

### 1. 空白の列挙(壁打ちプロトコル 1 手目)
材料全体に対して:
> 「この材料から、まだ誰も決めていないことを列挙しろ。埋めるな。」

出てきた項目を課題管理表(`docs/issue-log.md`・様式 = `templates/docs/issue-log.md`)に起票し、
各行に動詞を割り当てる: **観測する / 決める / 未決のまま載せる**。
観測コストが安いもの(30分で実 API を叩ける・実データを 1 件見られる)は、このあとの成果物作成より先に観測する。

### 2. 成果物の作成(この順で・薄い立ち上げでは作る分だけ)
各成果物は「AI が起草 → 人間が読む → 締めの申告(3 手目)→ 前提・仮定事項に転記」のループで作る。

1. **企画書** — 成功基準が観測可能な文になるまで
2. **業務フロー図**(As-Is / To-Be)— システム化範囲を枠で囲む
3. **機能一覧** — 各行に根拠(一次情報の参照)。優先度の第一キーは不確実な順
4. **用語集** — 中心概念の 1 文定義+反例。**実装時の型の枝は全部ここから生える**
5. **非機能要件** — 全項目を「要求あり(数値)/要求なし(理由)」のどちらかに

各成果物の締めは必ず:
> 「この文書で、材料に書かれていなくてあなたが自分で決めた箇所を全部挙げろ。
> 特に、名前・分類・『無い』の扱い・0 や空の意味・順序。」

### 3. 工程完了判定
チェックリスト(全部 yes で基本設計へ)。**判定対象は作った成果物だけ** — 薄い立ち上げで作らなかった
成果物は、課題管理表に「未作成」の 🔴 が 1 行あれば工程を出てよい:

- [ ] 成功基準は観測可能か(いつ・何を数えたら・いくつなら成功)
- [ ] 中心概念(このシステムが計算・管理するもの)は用語集で 1 文で定義されているか
- [ ] 機能一覧に根拠列が空の行は無いか(あれば課題管理表へ)
- [ ] 非機能要件に空欄は無いか(「要求なし(理由)」も記入のうち)
- [ ] 課題管理表の「未決」行のうち、観測コストが安いものは観測済みか
- [ ] **「未決」行に依存する記述が、課題番号の明記なしに成果物へ紛れていないか**

「未決」行がゼロである必要はない。未決に依存するものを未決と知らずに積むことだけが禁止。
