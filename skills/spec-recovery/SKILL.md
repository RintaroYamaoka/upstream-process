# /spec-recovery — 現行仕様の復元

既存システムに他人を入れる前の工程。古典で言う**現行仕様書**(as-built 文書)の復元。
一人+AI で作られたシステムには、AI が補完した決定が全レイヤに沈んでいる —
誰も決めていないのに決まった顔をしている型・名前・NULL・0 がある。
文書ゼロのままメンバーを入れると、メンバー(とそのAI)は沈んだ決定の上にさらに無音の補完を積む。

出力は `docs/現行仕様/` に、要件定義・基本設計と同じ様式(`templates/docs/` 参照)で書く。
**向きが逆なだけで成果物は同じ**: 立ち上げは空白を埋める前に列挙する、復元は埋まった空白を掘り起こす。

## 出所タグ(この skill 固有のルール)

全記述に 3 値のタグを付ける:

- **【記録】** — ADR・コミットメッセージ・既存文書・issue に根拠がある(人間が決めた痕跡)
- **【推定】** — コードからの推測。**AI が無音で埋めた可能性がある**
- **【不明】** — コードからも判断できない

## 進め方

### 1. 探査(read-only・コードを変更しない)
リポジトリ全体から列挙する: 機能(エンドポイント・画面・cron)/ テーブルと全カラム / 外部接続(API・環境変数の宛先)/
ドメイン語彙(型名・enum/union の枝・テーブル名・カラム名)。
規模が大きければ subagent / Workflow で並列に走らせてよい(read-only なので lane 隔離不要)。
既存の ADR・incident・コミットログも読み、【記録】の根拠台帳を先に作る。

### 2. 逆生成
docs/現行仕様/ に生成する(すべての行に出所タグ):

1. **用語集** — 型・テーブル・カラム名から語彙を起こす。**enum / union の各枝は「どれとどれが同じか」の主張** —
   その主張の根拠が【記録】に無ければ【推定】
2. **機能一覧** — 「誰が・何を・できる」+ 実際に使われているかの確認欄(アクセスログ・利用者に聞く)
3. **システム構成図+データ所有者表** — 実物確認の日付欄つき(復元でも実物を見る。コードが叩いている API が今も同じ形で返す保証はない)
4. **テーブル定義書** — 各カラムに 3 注記(保存か導出か / NULL の意味 / 0 の区別)。
   **導出できるのに保存されているカラム、意味の書けない NULL 許容は【推定】の典型**
5. **CRUD表** — 触られていないテーブル・何も触らない機能は削除候補として課題管理表へ
6. **外部インタフェース一覧**

### 3. 仕分けセッション(人間の工程・省略不可)
【推定】の各行を人間が読み、3 択に振る:

- **追認** → 根拠に「YYYY-MM-DD 本人追認」と書いて【記録】に昇格
- **要修正** → 課題管理表へ(直すのは別案件。この工程でコードを直さない)
- **未決** → 課題管理表へ

ここが本体。逆生成(1〜2)は機械の仕事、仕分け(3)だけが人間の仕事。
【推定】が数百行になっても、読むのはタグの付いた行だけで済む(全コードを読むのではない)。

### 4. チーム受け入れ準備
- **責任分担表** — 判断領域ごとの決定者。新メンバーに渡す判断領域を明記する
- **作業依頼書** — 最初の改修案件を様式どおりに書く(前提=現行仕様への参照 / 振る舞い / **あなたが決めること** / 未決課題)

## 完了判定

- [ ] 全成果物に出所タグが付いているか
- [ ] 【推定】の仕分けセッションは終わったか(残があるなら課題管理表に載っているか)
- [ ] 使われていない機能・触られないテーブルは課題管理表に載ったか
- [ ] **新メンバーが docs/現行仕様/ 一式と作業依頼書だけで着手できるか** —
  判定法: その一式を新しい AI セッションに貼って最初の案件を依頼したとき、AI の「自分で決めたこと」の申告が
  未決課題の範囲に収まるか(収まらない=文書に穴がある)
