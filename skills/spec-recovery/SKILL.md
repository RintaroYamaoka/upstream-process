# /spec-recovery — 現行仕様の復元(棚卸し)

一人+AI で作られたシステムには、AI が補完した決定が全レイヤに沈んでいる — 誰も決めていないのに
決まった顔をしている型・名前・NULL・0 がある。そして**それは docs にも沈んでいる**。
この工程の目的は 2 つ: **人間オラクルの理解負債の返済**(本人が全行を読んで追認する)と、
チーム受け入れ準備(新メンバーが着手できる文書一式)。前者が本体で、後者はその副産物。

## 第一規則: 根拠にできるのはコードと実データだけ

> この skill が必要になる状況とは、「docs が人間の理解を代表していない」状況のこと。
> docs を人間が古典的手法で書いたのなら、この skill は不要 — ただ読めばいい。
> だから**既存 docs から現行仕様を生成するのは循環**であり、AI の負債を「復元された仕様」に洗浄する行為。

- **根拠にしてよいもの**: コードの実測(型・スキーマ・エンドポイント・cron・実際の import 関係)/
  実データ・実挙動(本番データ・実 API レスポンス・実際に動かした結果)
- **既存 docs(ADR・incident・README・コミットメッセージ)は「主張」に降格**: 突合には使う
  (「ADR-0007 はこう主張しているが、コードと実データはどうか」— 食い違いは一級の発見)が、
  真実の根拠にはしない。AI が書いて人間が読まずに merge した doc は整形された推定にすぎない。
  **merge は署名だが、署名は理解ではない。**

## 読者の規則: 一次読者は人間

読むのは理解負債を返済する本人、次に新メンバー。AI 向けの前提パックは後工程(作業依頼書)の仕事。

- **散文で書く。** 表と ID の羅列にしない。各文書の冒頭に「この文書が答える問い」を 1 行置く
- **図は GitHub 上で描画される形式(mermaid)のみ。** 描画されない形式・外部ツール依存の図は禁止
- **古典の成果物セット(`references/成果物カタログ.md`)を全部生成する。**
  該当が無いものは「該当なし(理由)」を明記 — 欠落を無音にしない
  (画面があるのに画面遷移図が無い、は欠落であって省略ではない)

## 出所タグ(v0.2 で意味を変更): タグは信用度ではなく出所

信用を与える行為は人間の追認だけ。タグはその主張がどこから来たかだけを表す。

- **【コード】** — コードの実測から読み取った
- **【実測】** — 実データ・実挙動で確認した
- **【doc主張】** — 既存 docs がそう主張している(未検証)
- **【推定】** — 上のどれでもない推測

**タグは仕分けのアンカーでもある(1タグ=1主張=1判断)。** 散文には「行」が無いので、
仕分けの単位はタグで数える。だから生成時の制約:

- 主張を含む文には必ずタグを付け、**主張の直後**に置く(文末まとめ書きにしない)
- 1 つのタグに複数の主張をぶら下げない(仕分けできなくなる)。複合文は主張ごとにタグを分ける
- タグの無い文は主張を含んではならない(接続・説明のみ)

## 手順

### 1. 探査(read-only・コードと実データが先、docs は最後)
コードから列挙する: 機能(エンドポイント・画面・cron)/ テーブルと全カラム / 外部接続 /
ドメイン語彙(型名・enum/union の枝・テーブル名・カラム名)。実データ・実 API は可能な範囲で実測する。
規模が大きければ subagent / Workflow で並列に走らせてよい(read-only なので lane 隔離不要)。
**既存 docs はこの後に読み、全主張を突合台帳に載せる**。この工程の完了後に旧 docs は削除される(手順5)ので、
突合は「食い違いのリスト化」ではなく**全主張の処理**: 各主張が 吸収(実測と一致し現行仕様に写る)/
反証(実測と食い違う → 課題管理表へ)/ 失効(過去の経緯であり現在についての主張なし)のどれかになるまでやる。
未処理の主張が残っている限り、旧 docs は捨てられない。

### 2. 逆生成(人間向け散文+mermaid・セットに欠落なし)
`docs/現行仕様/` に、要件定義・基本設計と同じ成果物セットを生成する。全記述に出所タグ。
**セクション(見出し)は仕分けの承認単位**なので粒度を守る: 1 見出し=1 つの関心事、
中の主張はおおむね 5〜15 個。それを超えるなら小見出しに割る(承認印が形骸化しない大きさ)。
用語集では enum / union の各枝=「どれとどれが同じか」の主張として扱い、実データで裏が取れない枝は【推定】。
導出できるのに保存されているカラム、意味の書けない NULL 許容は【推定】の典型。

### 3. 仕分けセッション(人間の工程・**全主張**・省略不可)— 赤入れ+承認印方式(v0.4)
**出所タグを問わず全主張が対象。棚卸しに部分はない** —【コード】も【doc主張】も、
「コードがそうなっている」「doc がそう言っている」というだけで、本人が決めた・理解した保証はない。

方式は古典の文書レビューそのもの: **修正は本文への赤入れ、承認はセクション単位の印**。
主張 1 個ずつのマーカーは機械の都合を人間に押し付ける(初回転で棄却済み)。

0. **開始前に必ずベースラインを commit する**(赤入れの差分検出の基準。忘れると仕分けが消える)
1. セクション単位で読み、3 つの手を使い分ける:
   - **全部正しい** → セクション見出しに `【追認】` を 1 個(= そのセクション内で編集も Q も
     付けなかった主張の一括承認)
   - **間違っている** → **本文を直接書き換える**(マーカー不要。編集そのものが署名 = born-追認。
     ただしこの工程でコードは直さない — 直すのは docs だけ)
   - **決められない** → その主張に `【Q】`(唯一残る個別マーカー)

   人間が手で直す代わりに **AI に指示して直させた**場合、その主張の署名は一段弱くなる(下記の等級)。
   指示に含まれなかった細部は AI の補完で埋まっており、筆記は未検証だから。

### 署名の等級(オラクルがどこを通ったかの記録)

3 つとも**確定(終端状態)**であり、再確認の義務はどれにも無い。等級は出自の正直な記録であって、
宿題リストではない — 「指示で直させた箇所を読み直す優先度は低い」という本人の判断そのものを状態が記録する。

| 署名 | 意味 |
|---|---|
| `YYYY-MM-DD 本人訂正` | 本人がテキストを書いた |
| `YYYY-MM-DD 本人追認` | 本人がテキストを読んで承認した |
| `YYYY-MM-DD 本人指示(AI筆記)` | 本人が修正内容を決めて指示し、文面は AI が書いた(読み直しはしない) |

指示級の残余リスク(指示に含まれなかった細部は AI の補完で埋まっている)は承知の上での等級。
将来その主張に誤りが見つかったとき、故障箇所が「判断」か「筆記」かをこの署名が教えてくれる。
読む機会がたまたまあれば追認に日付更新してよい(任意・義務ではない)。
2. 終わったら AI に差分チェック+正規化をさせる。**この工程は台帳方式+検算必須**
   (散文で「突合しろ」と頼むだけでは実行 AI がマーカー置換だけの安い経路に逃げても検出できない。
   タグは grep で数えられる = この工程にはオラクルを付けられる。付けろ):

   - **ステップ1 列挙**: ベースライン以降の `git diff` の全 hunk に `D-01..` の ID を振り、
     台帳にする(ファイル・位置・編集要約)。マーカーのみの hunk と本文編集の hunk を分ける
   - **ステップ2 突合(本文編集を 1 件ずつ)**: 各 D-xx について該当コード(file:line を実際に読む)・
     実データと突合し、**根拠つきで** verdict を書く: **生成ミスの訂正**(実測と一致 →
     「YYYY-MM-DD 本人訂正」で確定)/ **仕様の変更**(実測と不一致 → docs が正でコードが追従 →
     課題管理表に採番・実装課題として起票)。根拠の無い verdict は禁止
   - **ステップ3 署名(セクション単位・本文を汚さない)**: 署名は**見出しに 1 個だけ**書く —
     `## 見出し 【確定 YYYY-MM-DD 本人追認】` の形式。**本文中の主張には署名を書き込まない**
     (文単位の署名は 863 個の署名増殖 = 人間可読性の破壊。初回転で棄却済み)。
     出所タグ(【コード】等)は本文にそのまま残す(将来の読者への出所情報)。
     見出し署名はその配下の全タグ(【推定】含む — 本人が読んで承認した)をカバーする。
     等級が混在するセクション(追認と指示が混ざる等)は**最も弱い等級を見出しに名乗る**(保守的)。
     どの主張がどの等級かの明細は D-xx 台帳(90-仕分けセッション.md)が持つ — 細粒度の真実は
     git 履歴+台帳にあり、docs 本文には置かない。`【Q】` は採番・起票して `→ Q-xx` を書き戻し
   - **ステップ4 検算(fail-loud)**: grep で数えて恒等式を確認 —
     **総タグ数 = 署名済みセクション配下のタグ数 + Q数 + 未処理セクション内のタグ数**。
     合わなければ取りこぼし。署名の無いセクションにタグが残っている=未処理として数える。
     なお本人の指示を受けて AI が修正した箇所の等級は「本人指示(AI筆記)」であり
     「本人訂正」「本人追認」を僭称しない
   - **ステップ5 報告**: D-xx 台帳全文 / 訂正:仕様変更:Q の内訳 / 検算結果 /
     未処理セクション残数(ゼロが完了条件)

### 4. チーム受け入れ準備
- **責任分担表** — 判断領域ごとの決定者。新メンバーに渡す判断領域を明記する
- **作業依頼書** — 最初の改修案件を様式どおりに書く(前提=確定済み行への参照 / 振る舞い /
  **あなたが決めること** / 未決課題)。**AI 向けに整形するのはここが初めて**

### 5. 旧 docs の廃棄(棚卸しの完成条件)
逆生成が完了した時点で **docs/現行仕様/ が唯一の正**になる。二本の正が並ぶと drift し、
AI が古い方を典拠として再引用する(tree に残っている限り、どのセッションの grep にも権威の顔で蘇る)。

- 旧 docs(ADR・incident・古い README 記述等のうち、突合台帳で処理済みのもの)を **HEAD から削除**する。
  `archive/` への退避はしない — AI の手の届く場所に権威が残る。**アーカイブは git 履歴そのもの**
  (人間は必要なとき掘れるが、AI のコンテキスト到達範囲からは消える。可逆だが無音で参照されない)
- 削除できない参照が残っている(コードやスクリプトが旧 doc のパスを読んでいる等)なら、
  それは逆生成の不完全のシグナル — 課題化して解消してから捨てる
- bootstrap 採用 repo では `docs/decisions/` 等のディレクトリ自体は残す(採用 marker・今後の記録の受け皿)。
  捨てるのは中身の旧文書

## 捨てたあとの不変条件(再負債の防止)

棚卸しは一回きりの移行儀式であり、以後この不変条件が立つ。これが無いと AI 製の文書がまた積もり、
数ヶ月後に二度目の棚卸しが要る(負債のリボ払い):

1. **現行仕様が唯一の正。** システムについての主張はここにだけ書く
2. **変更は作業依頼書経由で入り、完了時に現行仕様を改訂する**(古典の設計変更管理)。
   改訂されない変更は merge しない
3. **新しい決定文書は born-追認**: 書かれた瞬間に人間が根拠欄に日付付きで署名したものだけが確定。
   署名のない文書は【doc主張】のまま = 権威を持たない

## 完了判定

- [ ] **全行が仕分け済みか**(追認・Q-xx のどちらも付いていない行が残っていない = 理解負債の返済完了)
- [ ] 成果物セットに欠落がないか(「該当なし(理由)」も記入のうち)
- [ ] 図はすべて GitHub 上で描画されるか
- [ ] 突合台帳の全主張が処理済みか(吸収/反証/失効のどれかになっているか)
- [ ] **旧 docs を HEAD から削除したか**(削除できない参照が残っていない)
- [ ] **新メンバーが docs/現行仕様/ 一式と作業依頼書だけで着手できるか** —
  判定法: その一式を新しい AI セッションに貼って最初の案件を依頼したとき、AI の「自分で決めたこと」の申告が
  未決課題の範囲に収まるか(収まらない=文書に穴がある)
