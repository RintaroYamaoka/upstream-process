# /spec-recovery — 現行仕様の復元(棚卸し)

一人+AI で作られたシステムには、AI が補完した決定が全レイヤに沈んでいる — 誰も決めていないのに
決まった顔をしている型・名前・NULL・0 がある。そして**それは docs にも沈んでいる**。
この工程の目的は 2 つ: **人間オラクルの理解負債の返済**(本人が全行を読んで追認する)と、
チーム受け入れ準備(新メンバーが着手できる文書一式)。前者が本体で、後者はその副産物。

## 第一規則: 根拠にできるのはコードと実データだけ

> この skill が必要になる状況とは、「docs が人間の理解を代表していない」状況のこと。
> docs を人間が古典的手法で書いたのなら、この skill は不要 — ただ読めばいい。
> だから**既存 docs から現行仕様を生成するのは循環**であり、AI の負債を「復元された仕様」に洗浄する行為。

- **根拠にしてよいもの**: コードの実測(型・スキーマ・エンドポイント・cron・実際の import 関係)/
  実データ・実挙動(本番データ・実 API レスポンス・実際に動かした結果)
- **既存 docs(ADR・incident・README・コミットメッセージ)は「主張」に降格**: 突合には使う
  (「ADR-0007 はこう主張しているが、コードと実データはどうか」— 食い違いは一級の発見)が、
  真実の根拠にはしない。AI が書いて人間が読まずに merge した doc は整形された推定にすぎない。
  **merge は署名だが、署名は理解ではない。**

## 読者の規則: 一次読者は人間

読むのは理解負債を返済する本人、次に新メンバー。AI 向けの前提パックは後工程(作業依頼書)の仕事。

- **散文で書く。** 表と ID の羅列にしない。各文書の冒頭に「この文書が答える問い」を 1 行置く
- **図は GitHub 上で描画される形式(mermaid)のみ。** 描画されない形式・外部ツール依存の図は禁止
- **古典の成果物セット(`references/成果物カタログ.md`)を全部生成する。**
  該当が無いものは「該当なし(理由)」を明記 — 欠落を無音にしない
  (画面があるのに画面遷移図が無い、は欠落であって省略ではない)

## 出所タグ(v0.2 で意味を変更): タグは信用度ではなく出所

信用を与える行為は人間の追認だけ。タグはその主張がどこから来たかだけを表す。

- **【コード】** — コードの実測から読み取った
- **【実測】** — 実データ・実挙動で確認した
- **【doc主張】** — 既存 docs がそう主張している(未検証)
- **【推定】** — 上のどれでもない推測

## 手順

### 1. 探査(read-only・コードと実データが先、docs は最後)
コードから列挙する: 機能(エンドポイント・画面・cron)/ テーブルと全カラム / 外部接続 /
ドメイン語彙(型名・enum/union の枝・テーブル名・カラム名)。実データ・実 API は可能な範囲で実測する。
規模が大きければ subagent / Workflow で並列に走らせてよい(read-only なので lane 隔離不要)。
**既存 docs はこの後に読み**、主張と実測の食い違いを一覧化する(突合表)。

### 2. 逆生成(人間向け散文+mermaid・セットに欠落なし)
`docs/現行仕様/` に、要件定義・基本設計と同じ成果物セットを生成する。全記述に出所タグ。
用語集では enum / union の各枝=「どれとどれが同じか」の主張として扱い、実データで裏が取れない枝は【推定】。
導出できるのに保存されているカラム、意味の書けない NULL 許容は【推定】の典型。

### 3. 仕分けセッション(人間の工程・**全行**・省略不可)
**出所タグを問わず全行が対象。棚卸しに部分はない** —【コード】も【doc主張】も、
「コードがそうなっている」「doc がそう言っている」というだけで、本人が決めた・理解した保証はない。
人間が各行を 3 択に振る。追認だけが行を確定にする:

- **追認** → 「YYYY-MM-DD 本人追認」を付記(これが唯一の【確定】)
- **要修正** → 課題管理表へ(**この工程でコードは直さない** — 直し始めると棚卸しが終わらない)
- **未決** → 課題管理表へ

実務は 2 パスが速い(判断だけ人間・書類仕事は機械):
1. 読みながら行にマーカーだけ打つ — 追認は `【追認】`、課題送りは `【Q】`(1〜2文字、リズムを崩さない)
2. 最後に AI に一括正規化させる — `【追認】`→日付付き確定、`【Q】`→課題管理表に採番・起票して行に `→ Q-xx` を書き戻し、残数と件数を報告

### 4. チーム受け入れ準備
- **責任分担表** — 判断領域ごとの決定者。新メンバーに渡す判断領域を明記する
- **作業依頼書** — 最初の改修案件を様式どおりに書く(前提=確定済み行への参照 / 振る舞い /
  **あなたが決めること** / 未決課題)。**AI 向けに整形するのはここが初めて**

## 完了判定

- [ ] **全行が仕分け済みか**(追認・Q-xx のどちらも付いていない行が残っていない = 理解負債の返済完了)
- [ ] 成果物セットに欠落がないか(「該当なし(理由)」も記入のうち)
- [ ] 図はすべて GitHub 上で描画されるか
- [ ] docs の主張と実測の食い違い(突合表)はすべて課題管理表に落ちたか
- [ ] **新メンバーが docs/現行仕様/ 一式と作業依頼書だけで着手できるか** —
  判定法: その一式を新しい AI セッションに貼って最初の案件を依頼したとき、AI の「自分で決めたこと」の申告が
  未決課題の範囲に収まるか(収まらない=文書に穴がある)
