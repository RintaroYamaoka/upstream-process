# 成果物カタログ

各成果物について「どの決定を固定するか・完了基準・壁打ちでの作り方」を定義する。
様式(テンプレート)は `templates/docs/` 配下。成果物は対象プロジェクトの `docs/要件定義/` `docs/基本設計/`
(spec-recovery の場合は `docs/現行仕様/`)に置く。

## AI壁打ち共通プロトコル(全成果物に適用・3手)

1. **開始**: 材料(一次情報・前工程の成果物)を渡してこう指示する —
   「この材料から、**まだ誰も決めていないこと**を列挙しろ。埋めるな。」
   → 出てきた項目を課題管理表に起票する。
2. **起草**: 成果物のドラフトは AI が書いてよい。
3. **締め**: ドラフトに対してこう指示する —
   「この文書で、材料に書かれていなくて**あなたが自分で決めた**箇所を全部挙げろ。
   特に、名前・分類・『無い』の扱い・0 や空の意味・保存か導出か・順序。」
   → 挙がった箇所を文書の「前提条件・仮定事項」節に転記し、重要なものは課題管理表へ。

手を抜くなら 2 で抜く。1 と 3 を省くと、要件定義の段階で AI の推測が決定として文書に固定される。
**上流こそ AI の補完が最ももっともらしく見える場所**(動かして確かめられないため)。

## 課題への動詞は 3 つ(列挙した未決定にどう対処するか)

- **観測する** — 観測コストが安い(30分で API を叩ける・実データを 1 件見られる)なら、決めずに観測する
- **決める** — 決定者が自分なら決めて、根拠(観測した事実)と共に課題管理表に記録する
- **未決のまま載せる** — 決められないなら未決でよい。**ただしその課題に依存する成果物・実装を、課題番号の明記なしに作らない**(これが唯一のゲート)

# 要件定義の成果物

## 企画書

- **固定する決定**: なぜ作るか。何が観測されたら成功か。誰が関係者か。何を作らないか(スコープ外)
- **完了基準**: 成功基準が観測可能な文で書けている(「いつ・何を数えたら・いくつなら成功」)。スコープ外が明記されている
- **作り方**: 発注者(自分自身でも)への問い —「これが半年後に大成功だったとする。何がどうなっている?」を数字になるまで掘る
- 様式: `templates/docs/要件定義/企画書.md`

## 業務フロー図(As-Is / To-Be)

- **固定する決定**: いま業務がどう回っているか(As-Is)。システム導入後どう変わるか(To-Be)。**システム化の範囲(どこからどこまでを機械がやるか)**
- **完了基準**: 登場人物(人・外部システム)が全員レーンに載っている。As-Is が実際の業務担当者(または一次情報)で確認済み。To-Be でシステム化範囲が枠で囲まれている
- **作り方**: 一次情報(ヒアリング記録)は人間が全文読んでから。AI 要約から入らない(要約は語彙を平坦化し、業務の例外ケースが最初に消える)
- 様式: `templates/docs/要件定義/業務フロー.md`

## 機能一覧(機能要件一覧)

- **固定する決定**: 何を作るか。**どの順で作るか**(優先度)
- **完了基準**: 各機能が「誰が・何を・できる」の形で書けている。各行に根拠(一次情報のどこから来たか)がある。**根拠列が空の行は AI か自分の発明**なので課題管理表へ
- **作り方**: 優先度は「価値の大きい順」ではなく**「不確実な順」**を第一キーにする(最も不確実な仮説を最初に潰す。確実な器は後回しでも壊れない)
- 様式: `templates/docs/要件定義/機能一覧.md`

## 非機能要件

- **固定する決定**: 可用性・性能・セキュリティ・運用の要求水準。**そして「決めなくていいこと」**(社内 10 人利用なら性能要件は無い、と明記する)
- **完了基準**: 全項目が「要求あり(数値)」か「要求なし(理由)」のどちらかになっている。空欄が無い
- **作り方**: 空欄は「未決」であって「要求なし」ではない。両者を区別することがこの文書の存在理由
- 様式: `templates/docs/要件定義/非機能要件.md`

## 用語集

- **固定する決定**: **名前 = 「どれとどれが同じか」の主張**。ドメイン語彙の定義と、それが何で**ない**かの反例
- **完了基準**: 核心語すべてに 1 文の定義+反例(「予約は導線ではない。導線は入口、予約は結果」)+出所がある。プロジェクトの中心概念(このシステムが計算・管理するもの)が 1 文で定義されている
- **作り方**: AI に一次情報中の名詞を全部列挙させ、「同じものを指している語のペア/同じ語で違うものを指している箇所」を探させる。統合・分割の判断は人間
- **これが型定義の親**。実装時の enum / union の枝は全部ここから来る。用語集に無い枝が型に生えたら、それは AI の補完
- 様式: `templates/docs/要件定義/用語集.md`

## 課題管理表(全工程で継続)

- **固定する決定**: 何が未決かの台帳。決定したものは決定内容・根拠・決定者ごと記録(=決定記録を兼ねる)
- **完了基準**: 工程完了時に「未決」行がゼロである必要は**ない**。「未決」行に依存する後工程成果物が課題番号なしで存在しないこと
- **作り方**: 壁打ちプロトコル 1 手目の出力先。安く観測できる課題には「観測待ち」状態と観測方法を書く
- 様式: `templates/docs/要件定義/課題管理表.md`

# 基本設計の成果物

## システム構成図+データ所有者表

- **固定する決定**: **誰がどのデータを持っているか。どう取得するか。どれくらい新しいか**。関係者(外部サービス・API・人)は誰か
- **完了基準**: データ所有者表の**全行に「実物確認」の日付が入っている**。図が描けただけでは未完了。実物(実 API のレスポンス・実データ 1 件)を見ていない行は課題管理表の「観測待ち」
- **作り方**: 描いてから確かめるのではなく、確かめながら描く。外部 API は 30 分叩けば大抵の設計論争が消える
- 様式: `templates/docs/基本設計/システム構成図.md`

## テーブル定義書(ER図)

- **固定する決定**: 何を保存し、何を導出するか。「無い」はいつ起こるか(NULL の意味)。0 と「未観測」の区別
- **完了基準**: 各カラムに 3 つの注記がある — ①保存か導出か(**導出できる値を保存するなら理由必須**。保存した値は元データと乖離して嘘をつける) ②NULL 許容ならそれが業務上いつ起こるか ③数値カラムは 0 と「まだ無い」をどう区別するか
- **作り方**: 用語集からエンティティ候補を出す。AI にドラフトさせ、締めの申告(3手目)を必ず取る — テーブル設計は AI の補完が最も混入しやすい成果物
- 様式: `templates/docs/基本設計/テーブル定義書.md`

## 先行試作(プロトタイプ)— 文書ではなく実装

- **固定する決定**: 中心仮説(このシステムの存在理由の計算)が成立するか
- **完了基準**: 一番細いデータ経路 1 本が端から端まで通る(実データ入力 → 中心の計算 → CLI か HTML 1 枚の出力)。**UI・認証・管理画面を含まない**(器は骨ではない)
- **作り方**: これが最初の bootstrap 案件になる(実装なので管轄が移る)。基本設計の途中(構成図とテーブル定義の後、画面設計の**前**)に挟む
- 発注者・利用者に見せて所感を取る。ここで覆った場合の被害はほぼゼロ。画面を作った後で覆ると全損

## 画面一覧・画面遷移図

- **固定する決定**: どんな画面が要るか。各画面の目的
- **完了基準**: 各画面の目的が「誰が・何を・30秒以内にできる」の形で書けている。全画面が機能一覧のどれかに紐づく(紐づかない画面は誰の発明か?)
- **作り方**: **先行試作が生き残ってから描く**。見た目・並び・余白は描き込まない — それは実装担当(またはその場で決める人)の領分であり、ここで描き込むと相手の決定権を先取りする
- 様式: `templates/docs/基本設計/画面遷移図.md`

## CRUD表

- **固定する決定**: どの機能がどのテーブルを作成/参照/更新/削除するか(機能とデータの突合)
- **完了基準**: 全機能×全テーブルが埋まっている。**どの機能からも触られないテーブル、どのテーブルも触らない機能が残っていない**(残っていたらどちらかが発明か、機能・テーブルの漏れ)
- **作り方**: 機械的に埋まる表なので AI が作ってよい。読むのは交差チェックの結果だけ
- 様式: `templates/docs/基本設計/CRUD表.md`

## 外部インタフェース一覧

- **固定する決定**: 外部(他システム・API・ファイル連携)との境界の契約
- **完了基準**: 全行に実物確認の日付(実レスポンスを見た・レート制限を確認した)。**契約は相手の文書でなく相手の実挙動で確認する**
- 様式: `templates/docs/基本設計/外部インタフェース一覧.md`

## 責任分担表(RACI)

- **固定する決定**: **判断領域ごとに、誰が決めるか**。作業の分担ではなく判断の分担
- **完了基準**: 少なくとも次の領域に決定者が割当済み — ドメイン語彙と型 / データの保存・導出 / 画面の見せ方・並び / 何を先に作るか / 外部契約。**「その判断を間違えたとき、最初に気づけるのは誰か」が決定者**
- **作り方**: 安定した割当だけをこの表に書く(「画面の手触り=○○さん」)。案件ごとの割当は作業依頼書の「あなたが決めること」欄が主で、この表は従
- 様式: `templates/docs/基本設計/責任分担表.md`

## 作業依頼書

- **固定する決定**: 委託の境界 — 何が確定済みで、何を相手が決め、何がまだ未決か
- **完了基準**: ①前提の各行に出典(成果物名・課題番号)がある ②「実現してほしい振る舞い」が観測可能な文である ③**「あなたが決めること」が明記されている**(空欄は「全部こちらが決めた」の意味になり、相手を AI と同じ位置に置く) ④依存する未決課題が課題番号で列挙されている
- **受領側の締め**: 実装完了時、受領者(人間でも AI でも)に「依頼に書かれていなくて自分で決めたこと」を申告してもらい、依頼書の申告欄に転記する
- **変更管理**: 前提を覆すときは、その前提を引用している進行中の作業依頼書を先に洗い、依頼先に通知してから確定する(承認済みの依頼の下から前提を抜くと、相手の成果物が相手の落ち度の顔をして壊れる)
- 様式: `templates/docs/基本設計/作業依頼書.md`

# 現行仕様復元(spec-recovery)での使い方

上記と同じ成果物セットを `docs/現行仕様/` に**逆生成**する(欠落は「該当なし(理由)」で明示)。追加ルール:

1. **根拠にできるのはコードと実データだけ**。既存 docs(ADR 等)は「主張」に降格し突合対象にする
   — それ自体が AI に書かせた負債である可能性があるため(docs が信頼できるならこの skill は不要)
2. 全記述に**出所タグ**(信用度ではなく出所): **【コード】**(コードの実測)/ **【実測】**(実データ・実挙動)/
   **【doc主張】**(既存 docs の主張・未検証)/ **【推定】**(それ以外の推測)
3. **一次読者は人間**。散文で書く。図は GitHub で描画される mermaid のみ
4. 人間の仕分けは**全行**(タグを問わず)。追認だけが行を確定にする。棚卸しに部分はない

詳細は `skills/spec-recovery/SKILL.md`。
