# 工程と用語の地図

古典(共通フレーム / ウォーターフォールの標準工程)の全体像と、このプラグイン・bootstrap の管轄の対応。
人間のエンジニアと会話するとき、ここの用語をそのまま使えば通じる。

## V字モデル(工程の全体像)

各上流工程には、対応する検証工程が下流にある。**上流成果物とは「後で何と突き合わせて検証するかの宣言」**であり、
V字とはオラクルの配置図である。

```
企画(システム化計画) ──────────────→ 運用・効果測定
  要件定義 ─────────────────→ 受入テスト・総合テスト(システムテスト)
    基本設計(外部設計) ─────→ 結合テスト
      詳細設計(内部設計) ─→ 単体テスト
        実装
```

- **企画**: なぜ作るのか。成功の定義。↔ 運用に乗ってから効果で検証される
- **要件定義**: 何を作るのか(What)。発注者の言葉で書く。↔ 受入テスト(検証者は発注者/利用者)
- **基本設計(外部設計)**: 外から見える振る舞いの設計。画面・データ・外部接続。↔ 結合テスト
- **詳細設計(内部設計)**: 中身の設計。モジュール分割・処理方式。↔ 単体テスト
- **保守開発(エンハンス)**: 稼働後の改修。現行仕様の把握が前提。**要件が動く現場の定常運用は
  iteration-design(幹＋ベット)がここを担う** — V字を一度ずつ回すのでなく、薄いベットを連続で回して幹を増分改訂する

## 管轄の分界

| 工程 | 担当 |
|---|---|
| 企画〜要件定義〜基本設計(立ち上げ・移行) | **upstream-process**(このプラグイン)+ 人間の判断 |
| 稼働後の定常運用(要件が動く = 幹＋ベット) | 設計・計画・ねじれ除去 = **upstream-process**(iteration-design)/ 実装 = **project-bootstrap** |
| 詳細設計〜実装〜単体〜結合 | **project-bootstrap** + AI(hook で強制・詳細設計書は書かない。コードと test が代替) |
| 受入(要件定義の検証) | 人間。機械化不能。「動く」と「合っている」は別の検証 |

古典で詳細設計書・シーケンス図・関数仕様書が書かれたのは「安い実装者(人間)に間違えさせないための伝達装置」
だったから。実装者が AI になった時点でこの層の文書は死んだ。**残すのは決定を伝える文書、捨てるのは実装を指示する文書。**

## 用語対照表(古典 ↔ この環境)

| 古典の用語 | 意味 | この環境での対応物 |
|---|---|---|
| 課題管理表 | 未決事項の台帳。状態(未決/決定)と決定根拠を持つ | 未決マーカー(🔴)の正規版 = `docs/issue-log.md`。「未決の上に積まない」のゲート対象 |
| 前提条件・制約事項 | 文書冒頭の「この設計が依拠しているもの」の明記 | AI の補完申告(「仕様に無くて自分で決めたこと」)の転記先 |
| 責任分担表(RACI) | 判断領域ごとの決定者・相談先の割当 | 決定権表。lane owner の判断版 |
| 作業依頼書 | 前提+仕様を添えて作業を委託する文書 | 発注文(WO)。ベットフォルダ内に置き、「幹の該当セクション+1枚」で渡す |
| 設計変更管理 | 承認済み設計の変更時に影響範囲を洗い通知する手続き | 前提が覆ったら、それを引用する進行中の作業依頼書へ通知してから確定 |
| レビュー・承認 | 成果物単位の検査と署名 | **merge は署名だが理解ではない**。文書に権威を与えるのは born-追認(人間の署名)だけ |
| 先行試作(プロトタイプ) | 高リスク部分だけ先に小さく作って確かめる | walking skeleton。一番細いデータ経路 1 本・UI 無し |
| 現行仕様書 | 稼働中システムの as-built 文書 | 幹 = `docs/current-spec/`。更地では basic-design が初版を立て、既存では spec-recovery が逆生成する |
| 工程完了判定 | 次工程へ進んでよいかの関所 | 各 skill 末尾の完了基準チェックリスト |

## 古典からの意図的な逸脱(AI駆動での補正・最小限)

1. **「図が描けたら完了」を「実物を確認したら完了」に置き換える**(特にデータ関連)。
   古典の完了基準は文書の体裁で判定するため、自信満々で正反対の設計書が 2 枚通る。
   観測コストの安い事実(API を 30 分叩く・実データ 1 件を目で見る)は、決める前に観測する。
2. **画面設計を基本設計の主役から降ろす**。画面は「仮説が本物だった場合にだけ必要な器」。
   先行試作(データ経路 1 本)が生き残ってから描く。器を先に描くと器が仮説を守る側に回る。
3. **壁打ちプロトコル 3 手**(成果物カタログ冒頭)。AI は空白を質問でなく補完で返すため、
   「列挙しろ・埋めるな」と「自分で決めた箇所を申告しろ」を成果物作成の前後に固定で置く。
4. **定常モード(iteration-design)は V字を離れる**。要件が動く現場では上流を一度で固められない。
   古典の不変条件(未決を無音で埋めない)だけを引き継ぎ、全部入りの基本設計という様式は捨て、
   確度に儀式を比例させる幹＋ベット(Kanban 連続フロー × Shape Up のベット × 幹への born-追認)を回す。
   これは古典の再現ではなく**拡張** — 前提が安定した現場は 1〜3 の古典モードで足りる。

1〜3 は古典モード内の補正、4 は古典を離れる定常モード。それ以外は古典に従う。
さらなる最適化は、使って燃えてから足す(レトロスペクティブ経由)。

## 共通語彙(このプラグイン内の定義の正)

**同じ定義を複数ファイルで育てない** — プラグイン自身にも幹の規律を適用する(転記は幹が動いたとき追従しない)。
正の分界は 2 つだけ:

- **語彙(意味・関係・配置)の正 = この表**
- **成果物(完了基準・作り方)の正 = `references/artifact-catalog.md`**

skill・README・テンプレートの本文に現れる同種の記述は**使用点での短い再掲**であり、定義を育てる場所ではない。
語彙や規則を変えるときは、まずこの表(またはカタログ)を改訂し、再掲箇所を grep で追従させる。

| 語 | 定義 | 詳細の家 |
|---|---|---|
| 幹(現行仕様) | `docs/current-spec/`。システムについての主張の**唯一の正**。born-追認された delta の増分でのみ育つ(一括再導出はしない)。更地では basic-design が初版を立て、既存では spec-recovery が逆生成する | iteration-design / spec-recovery |
| ベット | 🟡 の前向きの賭け。**1 個 = 1 フォルダ** `docs/bets/<BET-slug>/`(ベット票+WO+局所メモが同居)。殺すときフォルダごと消す(アーカイブは git 履歴そのもの) | iteration-design ①⑤ |
| ベット票 | 6 欄 1 ページ(問題/賭け/棄却条件/測定の場所/appetite/実装者が決めてはいけない 1 点)。様式 `templates/docs/bets/bet.md` | iteration-design ① |
| 作業依頼書(WO) | 委託の境界文書。中核 4 節(前提/振る舞い/あなたが決めること/未決課題)。配置はベットフォルダ内 `WO-XX-<slug>.md`。前提欄は幹への**参照**(転記でなく)。様式 `templates/docs/bets/work-order.md` | basic-design 8 / iteration-design ② |
| 課題管理表 | 🔴 の台帳 `docs/issue-log.md`(工程をまたぐので工程フォルダに入れない)。様式 `templates/docs/issue-log.md` | requirements-definition |
| 確度 🔴🟡🟢 | 🔴 未決 = 台帳 1 行 / 🟡 暫定 = ベット票 1 ページ(未証明) / 🟢 確定 = 幹に増分。**🟡→🟢 を動かすのは merge でなく実測** | iteration-design |
| born-追認 | 文書は書かれた瞬間に人間が署名したものだけが確定。署名は**見出しに 1 個**(`## 見出し 【確定 YYYY-MM-DD 本人追認】`)。署名なき文書は【doc主張】のまま権威を持たない | spec-recovery / iteration-design ④a |
| 署名の等級 | `本人訂正`(本人が書いた)/ `本人追認`(読んで承認)/ `本人指示(AI筆記)`(内容を指示し文面は AI)。出自の正直な記録 | spec-recovery |
| 出所タグ | 【コード】【実測】【doc主張】【推定】。**信用度ではなく出所**。信用を与える行為は人間の追認だけ | spec-recovery |
| `🟡BET-XX` 印 | 幹の見出し署名に添える「出荷済み・幹に載っている・でも賭けは未証明」の印。④b の実測昇格で外す | iteration-design ④ |
