# upstream-process — AI と人間の認識のねじれを作らない上流工程

実装より上流の判断 — **何を決めるべきか** / **決めたことと AI が実際に作ったものを一致させ続けること** — を
AI との壁打ちで行う Claude Code プラグイン。核にあるのは一つの不変条件(下記): **未決定を無音で埋めない。**

この不変条件を **2 つのモード**で実装する:

- **古典モード**(立ち上げ・移行) — 数十年ぶん人間のチームを燃やして編纂された古典の成果物カタログを、
  空白の**検出器**として使う。requirements-definition / basic-design / spec-recovery。
- **定常モード**(要件が動く現場) — 古典は前提が安定した現場向けで churn に弱い。だから確度に儀式を比例させる
  **幹＋ベット**(Kanban 連続フロー × Shape Up のベット × XP の規律)を born-追認の幹の上で回す。iteration-design。
  **ここはもはや古典の再現ではなく、古典の不変条件を高速 churn 環境へ持ち込む拡張。**様式は捨て、不変条件だけ継ぐ。

## 不変条件(このプラグインの全体がこれの実行装置)

> **未決定事項は「観測される・決定される・課題管理表に載る」のいずれかであり、無音で埋まってはならない。AI・チームメンバー・リード自身の誰によっても。**

AI は空白(誰も決めていないこと)を質問ではなく「自信のある整形されたコード」で埋める。
人間のメンバーも、聞く場が無ければ埋めるしかない。古典の上流成果物は、その空白を
実装が始まる前に紙の上で可視化するための検出器として読み直せる(**古典モード**)。
**定常モード**では同じ空白を、出荷の継ぎ目(delta の追認)と本番の実測(棄却条件)で連続的に閉じる。

## project-bootstrap との分界

| | project-bootstrap | upstream-process |
|---|---|---|
| 管轄 | 詳細設計〜実装〜統合 | 企画〜要件定義〜基本設計 |
| 手段 | hook(fail-closed の強制・判断を含まない) | 手順+カタログ+壁打ち(判断を支える・強制しない) |
| 対象の誤り | リポジトリの中にオラクルがある誤り | リポジトリの中にオラクルが無い誤り |

判断を検出する gate は dead code になる(bootstrap 側 incident 済みの観測事実)。
だから上流は hook にしない。skill = 必要な時にコンテキストへ載る手順書、が正しい器。

## 成果物の読者は 3 者いる

1. 自分(と自分の AI セッション)
2. チームメンバー本人
3. **メンバーの AI セッション**(bootstrap 圏外。メンバーは渡された文書を自分の AI に貼る)

だから全成果物は「会議で読む体裁」ではなく「AI に貼ったとき前提として効く形式」で書く:
ID 付き・自己完結・コピペ可能な Markdown。作業依頼書の「前提」節は各成果物からの転記で成立する。

## 構成

```
skills/
  requirements-definition/  要件定義 — 立ち上げ時。何を先に決めるべきかの列挙から
  basic-design/             基本設計(外部設計) — データ実在確認・先行試作・作業依頼書まで
  spec-recovery/            現行仕様復元 — 既存システムにチームを入れる前の逆生成(移行儀式・一回きり)
  iteration-design/         反復設計(幹＋ベット) — 要件が動く現場の定常運用。確度に儀式を比例させ、
                            人間とAIの認識のねじれを出荷の継ぎ目で連続的に閉じる
references/
  成果物カタログ.md          各成果物が固定する決定・完了基準・壁打ちでの作り方
  工程と用語の地図.md        V字モデル・工程名・古典↔AI駆動の対照表
templates/docs/
  要件定義/                 企画書・業務フロー・機能一覧・非機能要件・用語集・課題管理表
  基本設計/                 システム構成図・画面遷移図・テーブル定義書・CRUD表・
                            外部インタフェース一覧・責任分担表・作業依頼書
  仮説/                     仮説票(iteration-design の 🟡 ベット・6欄:問題/賭け/棄却条件/測定の場所/appetite/決めてはいけない1点)
```

## 使い方

### どの skill をいつ使うか

| 状況 | skill |
|---|---|
| 新しく作る。実装前に何を決めるべきか分からない | `requirements-definition`(要件定義) |
| 要件定義が済んだ。チーム(人間・AI)に発注する前 | `basic-design`(基本設計) |
| 既にあるシステム(一人+AIで作った等)にチームを入れる / 理解負債を返済したい | `spec-recovery`(現行仕様復元) |
| 立ち上げは済み、幹が立っている。要件が動く中で定常的に回す(毎フェーズ作り直したくない) | `iteration-design`(反復設計・幹＋ベット) |

工程間のゲートは一つだけ: **未決の課題(課題管理表の「未決」行)に依存する成果物・実装を、課題番号の明記なしに作らない。**

### 新規プロジェクトの流れ(立ち上げ → 定常)

**立ち上げの規律(要件が動く現場ほど効く):** 不可逆の少数(スタック・コアドメイン・スキーマの形・外部契約)を
固め、可逆の多数(画面・機能・フロー)は 🔴 のまま課題管理表に載せる。**揮発性を幹の外へ押し出す**のが立ち上げの本質。

1. `/upstream-process:requirements-definition` — `docs/要件定義/`(企画書・業務フロー・機能一覧・非機能要件・用語集・課題管理表)。
   立ち上げでは全部を作り込まず薄く: 企画書 / 用語集の種 / 課題管理表 / 幹の設計。
2. **不可逆の幹だけ** `/upstream-process:basic-design` — `docs/基本設計/`(構成図・テーブル定義・先行試作・画面遷移図・CRUD表・外部IF・責任分担表・**作業依頼書**)。ここで幹(現行仕様の初版)が born-追認で立つ。⚠️ **フェーズごとに basic-design を回さない**(それが churn の元)。
3. 作業依頼書を渡して実装へ(ここから先は project-bootstrap の管轄)。
4. 要件が動き始めたら **`/upstream-process:iteration-design`(定常運用の主役)** → 下記「定常運用の流れ」。

### 既存システムの棚卸し(spec-recovery)の流れ

**生成(AI)→ 赤入れ(人間)→ 差分チェック(AI)→ 旧 docs 廃棄**、の4段。

**1. 生成** — 対象 repo の新しいセッションで `/upstream-process:spec-recovery`。
AI がコードと実データだけから `docs/現行仕様/` を逆生成する(既存 docs は根拠にしない — それ自体が負債の可能性があるため)。終わったら**必ずベースラインを commit**(次の赤入れを差分で検出するための基準)。

**2. 赤入れ(人間の工程・全セクション)** — エディタで `docs/現行仕様/` を上から読む。書く場所は **3パターンだけ**:

① **セクション全部OK → 見出しの行末に `【追認】`**
```markdown
## 顧客向け機能 【追認】
```
表や本文の中には何も書かない。この1個で、見出しから次の見出しまでの全主張を「読んだ、合っている」と宣言したことになる。

② **間違っている → 本文をその場で書き換える(マーカー不要)**
直した行為そのものが署名(あとで AI が git diff で拾う)。直した上で残りが OK なら見出しに `【追認】` も書く。**この工程でコードは直さない — 直すのは docs だけ。**

③ **迷う → その主張の出所タグの直後に `【Q】`**
```markdown
`src/app/xxx.tsx:36`【コード】【Q】
```

手の動き: セクションを読む → 直す所は直す → 迷いは `【Q】` → 見出しに `【追認】` → 次へ。
ほとんどのセクションは「読んで、見出しに `【追認】`」だけで終わる。

**3. 差分チェック(AI)** — 赤入れが終わったらセッションに貼る:

```
docs/現行仕様/ の仕分けをチェック・正規化しろ:
1. ベースライン commit 以降の git diff の各編集を、コードと実データに突合して2分類 —
   「生成ミスの訂正」(実測と一致)はそのまま確定し「YYYY-MM-DD 本人訂正」で署名 /
   「仕様の変更」(実測と不一致)は docs を正としてコード側の実装課題を課題管理表に採番・起票
2. セクション【追認】を、配下の全タグの「YYYY-MM-DD 本人追認」確定に展開
3. 【Q】を課題管理表に採番・起票して → Q-xx を書き戻し
4. 利用実態が「未使用」「未計測」「0行」のものを削除候補または観測課題として起票
   (記述の追認とは独立に)
5. 報告: 未処理セクションの残数(ゼロが完了)/ 追認・訂正・仕様変更・Q の件数
```

**4. 旧 docs の廃棄** — 突合台帳の全主張が処理済み(吸収/反証/失効)になったら、旧 docs を HEAD から削除する(`archive/` 退避は禁止。アーカイブは git 履歴そのもの)。以後は `docs/現行仕様/` が唯一の正: 変更は作業依頼書経由で完了時に改訂し、新しい決定文書は born-追認(署名なき文書は権威を持たない)。

### 定常運用の流れ(iteration-design)

立ち上げ(要件定義・基本設計)は済み、幹(`docs/現行仕様/`)が薄く立っている前提。要件が動く中で、
毎フェーズ全部入りを作り直さずに回す。方式は **Kanban 連続フロー × Shape Up のベット × XP の規律**を
追認された幹の上で回す「幹＋ベット」。目的は速度だけでなく **人間とAIの認識のねじれを溜めずに閉じる**こと。

1. **① ベット** — `/upstream-process:iteration-design`。🔴 を一つ選び、**6欄の仮説票**(`templates/docs/仮説/`)を
   *人間が*書く(問題 / 賭け / 棄却条件 / 測定の場所 / appetite / 実装者が決めてはいけない1点)。捨てる前提・born-未追認・幹と別 dir。
2. **② 発注** — 賭けを作業依頼書に割り、「あなたが決めること」で委譲する空白に名前を付ける(未決を無音にしない)。
3. **③ 実装** — 現場のフローで(bootstrap 採用なら hook が強制。**この skill は bootstrap に依存しない** — 無くても回る)。
4. **④a 継ぎ目追認** — merge 時、AI が実際に作った delta を読み、幹の該当セクションだけ born-追認で増分改訂
   (ここで閉じるのは「AIが意図どおり作ったか」= 内部妥当性。賭けはまだ 🟡)。
5. **④b 棄却条件の評価** — 出荷後、**測定の場所**の実測を棄却条件に照らす。**merge でなく実測が確度を上げる**:
   ヒット→殺す(⑤)/ 満たさず正のシグナル→ 🟢 昇格 / appetite 超過→再判断。
6. **⑤ 回路遮断** — 棄却されたら賭けを殺す(仮説票は破棄、git 履歴が保存)。**薄い賭けは殺せる。**

二度目の全 spec-recovery が要るなら、それはこのループが途切れたシグナル(spec-recovery は移行儀式+有界 drift 監査に限定)。

### ひとことチートシート

> **更地**: requirements-definition → basic-design(幹だけ) → 実装(bootstrap) → **iteration-design で定常** → 溜まったら spec-recovery で有界監査
> **既存**: spec-recovery で幹を立てる → iteration-design で定常
>
> 一貫する軸: **確度に儀式を比例**(🔴=課題1行 / 🟡=1ページ仮説票 / 🟢=幹に増分)・**幹は再構築でなく増分**・**merge でなく実測が 🟡→🟢 を動かす**。

### 出所タグの読み方

`【コード】` コードの実測 / `【実測】` 実データ・実挙動 / `【doc主張】` 既存 docs の主張(未検証) / `【推定】` 推測

**タグは出所であって信用度ではない。** 信用を与える行為は人間の追認(と赤入れ)だけ。

## 学習ルート(このプラグインの原典)

- 背骨(通読): 羽生章洋『はじめよう! 要件定義』技術評論社 — 「実装前に決めるべきことは何か・なぜか」だけを扱う薄い一冊
- カタログの原料(参照・通読しない): IPA『機能要件の合意形成ガイド』(無料PDF・画面/データモデル/外部IF等の分冊)
- 必要になったら: 羽生章洋『楽々ERDレッスン』/ Karl Wiegers『ソフトウェア要求』/ エヴァンス『ドメイン駆動設計』(用語集の本丸・参照扱い)
- 工程名の地図: 共通フレーム(references/工程と用語の地図.md に要約)
- **定常モード(iteration-design)の原典**: Ryan Singer『Shape Up』(ベット/appetite/circuit breaker) /
  Kent Beck『Extreme Programming』(TDD・CI・小さなリリース) / Kanban(WIP 上限で pull) /
  リーンスタートアップ(build-measure-learn = 棄却条件を先に書き実測で判定)。**古典と違い前提が動く現場向け** —
  古典の不変条件だけを引き継ぎ、V字の様式は捨てている。

使いながらレトロスペクティブで本プラグインに書き戻すこと。原典の継承を飛ばして
自分の事故の教訓だけで育てると「1年の経験を10回」に戻る。

## インストール

project-bootstrap と同じ方式で local plugin として追加する。
skill 単体を使いたい場合は `skills/<name>` を `~/.claude/skills/` に symlink してもよい
(その場合 `templates/` `references/` への相対参照が切れるので plugin 利用を推奨)。
