# upstream-process — AI駆動による古典的上流工程の再現

実装より上流(要件定義・基本設計)を、AI との壁打ちで行うための Claude Code プラグイン。
数十年ぶん人間のチームを燃やして編纂された古典の成果物カタログを、そのままの用語・様式で使う。
AI 駆動への最適化は最小限(壁打ちプロトコル 3 手のみ)にとどめ、まず古典を忠実に再現する。

## 不変条件(このプラグインの全体がこれの実行装置)

> **未決定事項は「観測される・決定される・課題管理表に載る」のいずれかであり、無音で埋まってはならない。AI・チームメンバー・リード自身の誰によっても。**

AI は空白(誰も決めていないこと)を質問ではなく「自信のある整形されたコード」で埋める。
人間のメンバーも、聞く場が無ければ埋めるしかない。古典の上流成果物は、その空白を
実装が始まる前に紙の上で可視化するための検出器として読み直せる。

## project-bootstrap との分界

| | project-bootstrap | upstream-process |
|---|---|---|
| 管轄 | 詳細設計〜実装〜統合 | 企画〜要件定義〜基本設計 |
| 手段 | hook(fail-closed の強制・判断を含まない) | 手順+カタログ+壁打ち(判断を支える・強制しない) |
| 対象の誤り | リポジトリの中にオラクルがある誤り | リポジトリの中にオラクルが無い誤り |

判断を検出する gate は dead code になる(bootstrap 側 incident 済みの観測事実)。
だから上流は hook にしない。skill = 必要な時にコンテキストへ載る手順書、が正しい器。

## 成果物の読者は 3 者いる

1. 自分(と自分の AI セッション)
2. チームメンバー本人
3. **メンバーの AI セッション**(bootstrap 圏外。メンバーは渡された文書を自分の AI に貼る)

だから全成果物は「会議で読む体裁」ではなく「AI に貼ったとき前提として効く形式」で書く:
ID 付き・自己完結・コピペ可能な Markdown。作業依頼書の「前提」節は各成果物からの転記で成立する。

## 構成

```
skills/
  requirements-definition/  要件定義 — 立ち上げ時。何を先に決めるべきかの列挙から
  basic-design/             基本設計(外部設計) — データ実在確認・先行試作・作業依頼書まで
  spec-recovery/            現行仕様復元 — 既存システムにチームを入れる前の逆生成
references/
  成果物カタログ.md          各成果物が固定する決定・完了基準・壁打ちでの作り方
  工程と用語の地図.md        V字モデル・工程名・古典↔AI駆動の対照表
templates/docs/
  要件定義/                 企画書・業務フロー・機能一覧・非機能要件・用語集・課題管理表
  基本設計/                 システム構成図・画面遷移図・テーブル定義書・CRUD表・
                            外部インタフェース一覧・責任分担表・作業依頼書
```

## 使い方

### どの skill をいつ使うか

| 状況 | skill |
|---|---|
| 新しく作る。実装前に何を決めるべきか分からない | `requirements-definition`(要件定義) |
| 要件定義が済んだ。チーム(人間・AI)に発注する前 | `basic-design`(基本設計) |
| 既にあるシステム(一人+AIで作った等)にチームを入れる / 理解負債を返済したい | `spec-recovery`(現行仕様復元) |

工程間のゲートは一つだけ: **未決の課題(課題管理表の「未決」行)に依存する成果物・実装を、課題番号の明記なしに作らない。**

### 新規プロジェクトの流れ

1. `/upstream-process:requirements-definition` — 成果物は `docs/要件定義/`(企画書・業務フロー・機能一覧・非機能要件・用語集・課題管理表)
2. 完了基準を満たしたら `/upstream-process:basic-design` — 成果物は `docs/基本設計/`(構成図・テーブル定義・先行試作・画面遷移図・CRUD表・外部IF・責任分担表・**作業依頼書**)
3. 作業依頼書が書けたら実装へ(ここから先は project-bootstrap の管轄)

### 既存システムの棚卸し(spec-recovery)の流れ

**生成(AI)→ 赤入れ(人間)→ 差分チェック(AI)→ 旧 docs 廃棄**、の4段。

**1. 生成** — 対象 repo の新しいセッションで `/upstream-process:spec-recovery`。
AI がコードと実データだけから `docs/現行仕様/` を逆生成する(既存 docs は根拠にしない — それ自体が負債の可能性があるため)。終わったら**必ずベースラインを commit**(次の赤入れを差分で検出するための基準)。

**2. 赤入れ(人間の工程・全セクション)** — エディタで `docs/現行仕様/` を上から読む。書く場所は **3パターンだけ**:

① **セクション全部OK → 見出しの行末に `【追認】`**
```markdown
## 顧客向け機能 【追認】
```
表や本文の中には何も書かない。この1個で、見出しから次の見出しまでの全主張を「読んだ、合っている」と宣言したことになる。

② **間違っている → 本文をその場で書き換える(マーカー不要)**
直した行為そのものが署名(あとで AI が git diff で拾う)。直した上で残りが OK なら見出しに `【追認】` も書く。**この工程でコードは直さない — 直すのは docs だけ。**

③ **迷う → その主張の出所タグの直後に `【Q】`**
```markdown
`src/app/xxx.tsx:36`【コード】【Q】
```

手の動き: セクションを読む → 直す所は直す → 迷いは `【Q】` → 見出しに `【追認】` → 次へ。
ほとんどのセクションは「読んで、見出しに `【追認】`」だけで終わる。

**3. 差分チェック(AI)** — 赤入れが終わったらセッションに貼る:

```
docs/現行仕様/ の仕分けをチェック・正規化しろ:
1. ベースライン commit 以降の git diff の各編集を、コードと実データに突合して2分類 —
   「生成ミスの訂正」(実測と一致)はそのまま確定し「YYYY-MM-DD 本人訂正」で署名 /
   「仕様の変更」(実測と不一致)は docs を正としてコード側の実装課題を課題管理表に採番・起票
2. セクション【追認】を、配下の全タグの「YYYY-MM-DD 本人追認」確定に展開
3. 【Q】を課題管理表に採番・起票して → Q-xx を書き戻し
4. 利用実態が「未使用」「未計測」「0行」のものを削除候補または観測課題として起票
   (記述の追認とは独立に)
5. 報告: 未処理セクションの残数(ゼロが完了)/ 追認・訂正・仕様変更・Q の件数
```

**4. 旧 docs の廃棄** — 突合台帳の全主張が処理済み(吸収/反証/失効)になったら、旧 docs を HEAD から削除する(`archive/` 退避は禁止。アーカイブは git 履歴そのもの)。以後は `docs/現行仕様/` が唯一の正: 変更は作業依頼書経由で完了時に改訂し、新しい決定文書は born-追認(署名なき文書は権威を持たない)。

### 出所タグの読み方

`【コード】` コードの実測 / `【実測】` 実データ・実挙動 / `【doc主張】` 既存 docs の主張(未検証) / `【推定】` 推測

**タグは出所であって信用度ではない。** 信用を与える行為は人間の追認(と赤入れ)だけ。

## 学習ルート(このプラグインの原典)

- 背骨(通読): 羽生章洋『はじめよう! 要件定義』技術評論社 — 「実装前に決めるべきことは何か・なぜか」だけを扱う薄い一冊
- カタログの原料(参照・通読しない): IPA『機能要件の合意形成ガイド』(無料PDF・画面/データモデル/外部IF等の分冊)
- 必要になったら: 羽生章洋『楽々ERDレッスン』/ Karl Wiegers『ソフトウェア要求』/ エヴァンス『ドメイン駆動設計』(用語集の本丸・参照扱い)
- 工程名の地図: 共通フレーム(references/工程と用語の地図.md に要約)

使いながらレトロスペクティブで本プラグインに書き戻すこと。原典の継承を飛ばして
自分の事故の教訓だけで育てると「1年の経験を10回」に戻る。

## インストール

project-bootstrap と同じ方式で local plugin として追加する。
skill 単体を使いたい場合は `skills/<name>` を `~/.claude/skills/` に symlink してもよい
(その場合 `templates/` `references/` への相対参照が切れるので plugin 利用を推奨)。
